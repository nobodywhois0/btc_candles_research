§02 · Organización mínima

Cinco carpetas, no quince módulos

edge-research/ ├── data/ # raw + processed + features + datasets, particionado por fecha — sin subdividir en 5 carpetas separadas ├── lib/ # código compartido: ingestion, validation, features P0, stats, report — funciones puras, sin clases abstractas ├── research/ # una carpeta por hipótesis — ver §03 ├── runs/ # resultados versionados de cada ejecución (JSON + HTML + figuras) └── notebooks/ # exploración libre — nunca dependencia de lib/ ni de research/

Lo que desaparece respecto a v0.4: config/ (vive dentro de cada run.py o en un único lib/config.py), tests/ como directorio propio (los tests viven junto a la función que prueban), docs/ (el README de cada hipótesis es la documentación), cli/ (se ejecuta python research/001_survival/run.py directo), scripts/ como categoría separada de notebooks/.

§03 · Desarrollo guiado por hipótesis

Una carpeta, una pregunta, una respuesta

research/ ├── 001_survival/ │ ├── hypothesis.md # 1 página: pregunta, mecanismo esperado, predicción falsable, criterio de decisión │ ├── run.py # único script: carga → calcula → testea → guarda │ ├── results/ │ │ ├── report.html # generado por lib/report.py │ │ ├── figures/ # PNG/HTML de las gráficas de §13 │ │ └── summary.json # métricas clave + estado — alimenta el tablero de §14 │ └── README.md # notas del investigador: qué se encontró, dudas, próximo paso ├── 002_shape/ # misma estructura exacta ├── 003_microstructure/ ├── 004_orderflow/ └── 005_entropy/

Regla dura de independencia: ninguna carpeta de research/ importa de otra — solo de lib/ (funciones puras) y de data/ (solo lectura). Agregar la hipótesis 006 es copiar la carpeta 001 como plantilla y cambiar tres cosas: la pregunta, el cálculo específico, el test estadístico. Nunca toca lib/ salvo que necesite una función genuinamente nueva y reutilizable.

§04 · Roadmap científico

Cada semana termina con una pregunta respondida, no con código escrito

La infraestructura de una semana (p. ej. el script de ingestión de la semana 1) es la respuesta a la pregunta de esa semana, no un prerrequisito separado. Ninguna semana existe solo para "construir algo".

  1. Semana 1
    ¿Son confiables los datos? Ingestion de klines 1m/5m (2-3 años) + validación de 3 reglas + reporte de calidad: % gaps, duplicados, inconsistencias OHLC.
  2. Semana 2
    ¿Es el mercado, a 5 minutos, indistinguible de un paseo aleatorio? Experimento cero: runs test, variance ratio, BDS sobre la serie de signos. Gate real, no trámite — ver §15.
  3. Semana 3
    ¿Existe memoria simple entre velas? Autocorrelación de signos, cadenas de Markov orden 1-2.
  4. Semanas 4-5
    ¿La supervivencia del color contiene información? 001_survival completo — el experimento más importante del proyecto, ver §13.
  5. Semanas 6-7
    ¿La forma interna de la vela (1m→5m) predice la siguiente? 002_shape: run-length, posición del último flip, clustering simple de trayectorias.
  6. Semana 8
    Checkpoint de decisión #1. Con lo aprendido en semanas 1-7, ¿hay base para seguir? Se actualiza el tablero completo (§14).
  7. Semanas 9-10
    ¿La cinemática intra-vela aporta señal incremental? 003_microstructure: eficiencia del camino, tiempo de ocupación, velocidad/aceleración.
  8. Semanas 11-12
    ¿El flujo de órdenes predice dirección? 004_orderflow: taker buy ratio, delta, OFI aproximado.
  9. Semana 13
    ¿El régimen modula la señal de las hipótesis anteriores? 005_entropy: entropía de Shannon, Hurst, changepoint — como variable de interacción, no como hipótesis aislada.
  10. Semana 14
    Checkpoint de decisión #2. ¿Hay evidencia suficiente acumulada para pasar a un modelo combinado? Aplicar §08 con rigor.
  11. Semanas 15-16
    ¿Un modelo combinado supera a la mejor hipótesis individual? Regresión logística sobre las features validadas — solo si semana 14 dice "sí".
  12. Semanas 17-18
    ¿La señal sobrevive validación walk-forward y calibración honesta?
  13. Semana 19
    ¿LightGBM supera significativamente a la regresión logística? Solo si 15-18 mostró algo que valga la pena capturar de forma no lineal.
  14. Semana 20
    ¿El EV neto del payout real del producto Up/Down es positivo con significancia? Simulación económica final.
  15. Semana 21
    Checkpoint de decisión final — go/no-go.
  16. Semanas 22-24
    Buffer. Si hay señal: profundizar la línea ganadora. Si no: escribir el postmortem honesto (§15), no maquillarlo.

§06 · Experimentos permitidos

El filtro: ¿puede cambiar la decisión?

Un experimento entra a la lista solo si es posible escribir de antemano qué resultado numérico llevaría a abandonar y cuál a continuar. Si no se puede escribir esa frase antes de correrlo, se elimina.

Entran

Experimento cero (eficiencia de mercado), tabla de supervivencia + Kaplan-Meier, estructura interna/shape, taker buy ratio direccional, modelo combinado walk-forward (AUC/Brier vs. nulo), simulación de EV económico final.

No entran

Patrones de velas japonesas (ya se sabe, desde v0.2, que no van a mover la aguja — no hace falta gastar una semana en redescubrirlo). Indicadores técnicos clásicos como prueba aislada (mismo argumento). Macro/on-chain (horizonte desalineado, ya descartado en v0.1-v0.2). Cualquier variante "por curiosidad" sin criterio de decisión escrito de antemano.

§07 · Modelos permitidos

Tablas de frecuencia → regresión logística → LightGBM. Nada más, todavía.

  • Tablas de frecuencia / Kaplan-Meier: riesgo de sobreajuste esencialmente nulo, máxima interpretabilidad, responden la forma más cruda posible de la pregunta.
  • Tests estadísticos: formalizan significancia, controlan la tasa de falsos positivos con corrección por comparaciones múltiples.
  • Regresión logística: el modelo multivariante más barato que existe, coeficientes interpretables. Regla de la escalera (v0.1 §06): si esto no supera al nulo, es muy improbable que algo más complejo lo haga — es la inversión de mayor valor por hora de las primeras 16 semanas.
  • LightGBM, y solo LightGBM: la única concesión a no-linealidad permitida, y solo después de que la regresión logística muestre algo. Entrena en segundos con este volumen de datos, buena regularización por defecto, sin necesidad de GPU. No se prueban XGBoost/CatBoost/Random Forest en paralelo — son marginalmente redundantes entre sí y comparar tres para elegir uno es tiempo de ingeniero que no responde una pregunta nueva de mercado.
Explícitamente prohibido en 6 meses

LSTM, Transformer, TFT, cualquier red neuronal secuencial: infraestructura pesada, sin justificación previa de no-linealidad/efecto de secuencia que logística+LightGBM no puedan ya capturar. Se reconsideran solo si semanas 15-19 muestran evidencia específica de que la insuficiencia está en la forma funcional, no en la ausencia de señal.

§08 · Métrica de éxito del proyecto

Criterios cuantitativos, fijados antes de ver resultados

Continuar (tras checkpoint semana 14 o 21) si TODO lo siguiente se cumple
  • Al menos una hipótesis individual sobrevive un permutation test con p<0.01 tras corrección FDR por el número total de hipótesis probadas.
  • El tamaño del efecto es económicamente relevante, no solo estadístico: mejora relativa de Brier Score >2% sobre el modelo nulo, sostenida en ≥70% de los folds walk-forward.
  • El EV neto del payout real del producto Up/Down es positivo con un intervalo de confianza (bootstrap) que excluye cero.
  • La señal es estable en al menos 3 años/regímenes distintos — no es un artefacto de un solo periodo.
Abandonar si CUALQUIERA de lo siguiente ocurre
  • Ninguna hipótesis sobrevive el permutation test corregido por FDR.
  • Sobrevive pero el efecto es económicamente irrelevante (mejora de Brier <0.5%, o EV neto no distinguible de cero incluso en el mejor caso).
  • La señal es inestable entre folds/regímenes — aparece en un año y desaparece en validación.
Estado intermedio — no se fuerza una decisión prematura

Requiere más datos: alguna hipótesis muestra p<0.05 sin sobrevivir la corrección estricta, y el subconjunto relevante (p. ej. un régimen específico de volatilidad) no tiene tamaño de muestra suficiente para un test con potencia adecuada. Se documenta como abierto, se define qué dato adicional resolvería la ambigüedad, y se agenda — no se decide con datos insuficientes solo por presión de calendario.

§10 · Qué NO construir todavía

Cada renglón, con la razón exacta por la que retrasa el aprendizaje

  • Plugin registry / auto-discovery — resuelve un problema (descubrir 500 hipótesis) que no existe con 5. Tiempo de ingeniería que no responde ninguna pregunta de mercado.
  • Airflow / Prefect / Dagster — orquestan un pipeline que, en 6 meses, un humano ejecuta a mano una vez por semana. El scheduler es el propio investigador.
  • Redis — no hay streaming en tiempo real que lo justifique en este horizonte.
  • Kubernetes / arquitectura distribuida — el dataset completo cabe en RAM de una laptop (Fase 2 §14.4). El cuello de botella es entender el mercado, no escalar cómputo.
  • Dashboards — resuelven "¿cómo va todo, en vivo?"; un board.md actualizado una vez por semana (§14) cumple la misma función a coste cero.
  • APIs / interfaces web — no hay un segundo usuario en 6 meses. Construir una API es diseñar para alguien que no existe todavía.
  • Optimización prematura (caching sofisticado, paralelización, GPU) — el cómputo completo toma minutos; cada hora aquí es una hora no dedicada a una hipótesis.
  • Versionado exhaustivo de 4 hashes — se sustituye por git commit + copia de config (§11), suficiente hasta que haya algo valioso que proteger de irreproducibilidad.
  • Multi-modelo (7 librerías) — ver §07; fragmenta esfuerzo comparando alternativas que rendirán de forma similar.
  • Documentación automática — el research/ folder y su README ES la documentación.
  • Testing de 4 capas — solo unitarios de las funciones de cálculo más fáciles de equivocar silenciosamente; nada de regression suite hasta que haya resultados que valga la pena proteger.
  • Meta-labeling arquitectónico — es una función condicional cuando exista un modelo primario con señal, no un módulo de antemano.
§11 · Decisiones irreversibles

Una decisión es irreversible si cambiarla exige reprocesar histórico o invalida resultados ya obtenidos

Decidir bien desde el día 1

Formato de almacenamiento (Parquet) — cambiarlo después obliga a reprocesar todo el histórico acumulado. Representación temporal (UTC, ancla en el cierre de vela, no en la apertura) — un error de convención descubierto tarde invalida silenciosamente todos los experimentos hechos hasta ese punto. Esquema mínimo del grano (vela de 5m + sub-barras de 1m) — cambiar el grano después de 20 hipótesis construidas sobre él es empezar de nuevo. Definición del target y el feed de precio (spot/perp/feed real del bróker) — cambiarlo invalida cualquier cálculo de EV ya hecho.

Se puede decidir después, sin miedo

Organización interna de lib/ (nombres de archivo). Motor de gráficas (matplotlib vs. plotly) — cosmético. Reporte como template HTML o notebook exportado — no afecta ningún resultado científico. Versionado más allá de git básico (DVC/hashing) — se añade después sin reprocesar nada, siempre que Parquet ya esté bien fijado. Sistema de logging. CLI vs. scripts directos.

§12 · Riesgos del proyecto — sin cortesías

Por qué esta investigación podría fracasar

  • El riesgo dominante es que simplemente no hay edge. BTC/USDT en Binance a 5 minutos es uno de los mercados más arbitrados de todo el cripto: miles de bots, market makers profesionales, el propio exchange con incentivos de estabilidad. El prior razonable de encontrar un edge grande y persistente es bajo — esto no es un riesgo de ejecución, es un riesgo de la pregunta misma. Hay que estar dispuestos a que la respuesta correcta sea "no".
  • Un p-valor significativo no es un edge operable. El umbral relevante es económico (EV neto de p*), no estadístico — un efecto real pero minúsculo puede no sobrevivir comisiones y spread del producto Up/Down real.
  • Sesgo de selección y supervivencia de resultados. Con 5+ hipótesis y docenas de features, encontrar "algo" por puro azar es probable sin corrección por comparaciones múltiples. Reportar solo lo que "funcionó" es el sesgo más peligroso porque es invisible para el propio investigador — mitigación: este documento es el pre-registro; toda hipótesis probada se reporta, funcione o no.
  • P-hacking por iteración, agravado por ser un equipo de una persona. Ajustar umbrales o ventanas hasta que "funcione" después de ver resultados es indistinguible, en la práctica, de inventar el hallazgo — y nadie más audita las decisiones de ajuste en un equipo de uno.
  • No estacionariedad. BTC 2020-21 no es BTC 2023-24; seis meses de calendario pueden no cubrir régimen suficiente para distinguir edge real de artefacto de un periodo particular.
  • Calidad de datos silenciosa. Un error en la reconstrucción de sub-barras de 1m o una convención de timestamp ligeramente incorrecta puede generar una señal falsa que en realidad es leakage — el bug más peligroso es el que hace que los resultados se vean demasiado buenos.
  • Fatiga de rigor. Con presión de tiempo, existe riesgo conductual real de relajar la disciplina a mitad de camino ("no corrijo por comparaciones múltiples esta vez", "me salto el permutation test porque ya se ve claro"). Este documento lo trata como regla dura, no como sugerencia.
  • Riesgo de que las reglas del producto cambien. El punto de equilibrio p* del bróker Up/Down puede moverse (comisión, payout) de forma que invalide un análisis de EV que era correcto cuando se hizo.
§13 · El experimento más importante

Si solo hubiera un mes: supervivencia del color

Es el que ya señalaban v0.2 y v0.3 como la línea de mayor valor esperado por unidad de esfuerzo, y el único que puede redefinir el problema de negocio (cuándo apostar dentro de la vela, no solo en qué dirección).

Datos

Histórico completo disponible, mínimo 2-3 años, klines de 1m del feed real del producto (spot/perp según lo fijado en §11), reconstruyendo cada vela de 5m como 5 sub-barras.

Qué se calcula

Para cada vela histórica, el lado del open en cada marca T∈{30,60,...,270}s, y si coincidió con el color final. De ahí: (a) tabla de frecuencia P(cierra en el mismo lado | lado en T) por cada T; (b) curvas de Kaplan-Meier del tiempo hasta que el color se fija, estratificadas por tercil de ATR; (c) modelo de Cox, solo si (a) y (b) ya muestran algo.

Gráficos

  • P(cierra verde | verde en T) vs. T, con banda de confianza de Wilson.
  • Curvas Kaplan-Meier por tercil de volatilidad, superpuestas.
  • Histograma de time_to_color_lock_sec.
  • Heatmap T × régimen de volatilidad → probabilidad de supervivencia.

Estadísticos

Diferencia entre P(verde|verde en T) y la frecuencia base incondicional, con IC de Wilson por T; log-rank test entre curvas KM de distintos regímenes; permutation test barajando la asignación de "lado en T" preservando estructura temporal; tamaño de muestra efectivo por T (los T cercanos al cierre tienen menos observaciones aún "indecisas").

El punto más sutil, y el más importante de todo el documento

Incluso un mercado puramente aleatorio (random walk sin ninguna estructura) produce una curva de "supervivencia" ascendente con T — simplemente porque queda menos tiempo para que algo cambie. El benchmark correcto no es 50% plano: es la curva que produciría un movimiento browniano puro con la misma volatilidad histórica, simulada explícitamente. Cualquier resultado positivo debe compararse contra ese benchmark simulado antes de interpretarse como información real, no contra una línea recta ingenua.

Decisión según resultado

  • P(verde|verde en T=240s) sustancialmente por encima del benchmark de random walk (no de 50%), con IC estrecho y significativo tras permutation test → señal fuerte, continuar con fuerza hacia 002_shape y 003_microstructure.
  • Por encima del benchmark pero con margen modesto → evidencia débil; extender histórico y revisar si el efecto se concentra en algún régimen antes de decidir.
  • Indistinguible del benchmark de random walk para todo T → evidencia fuerte para abandonar esta línea específica (no necesariamente el proyecto completo, hasta agotar 002 y 003).

§15 · Postmortem — pregunta final

Seis meses después: el proyecto fracasó. ¿Por qué?

Qué pasó. Ninguna de las cinco hipótesis centrales sobrevivió corrección por comparaciones múltiples con un tamaño de efecto económicamente relevante. La curva de supervivencia del color —la apuesta de mayor prioridad de todo el programa— resultó, una vez comparada contra el benchmark correcto de random walk simulado con la misma volatilidad (no contra 50% plano), indistinguible de lo que produce un movimiento browniano puro. La "señal" que entusiasmó al equipo en las primeras semanas era exactamente el artefacto mecánico que este mismo documento advertía en §13, y en el optimismo inicial se le dio más crédito del que merecía antes de construir el benchmark correcto.

Qué señales ignoramos. El experimento cero (semana 2) ya daba una pista: ninguno de los tres tests de eficiencia rechazó la hipótesis de martingala con robustez, y se trató como un trámite protocolario en vez de como una actualización bayesiana fuerte hacia "esto va a ser difícil". La señal de order flow imbalance (semanas 11-12) mostraba un patrón real pero minúsculo — al pasarlo por la simulación de EV neto del payout real, el resultado ya era negativo en la semana 12, y aun así se dedicaron tres semanas más a "afinar" el modelo buscando que el número cambiara de signo: exactamente el patrón de p-hacking que §12 advertía evitar. Más de una vez el investigador notó que estaba comparando resultados después de ver datos de test, ajustando un umbral "solo un poco", sin documentarlo como lo que era: una violación de la disciplina de validación temporal que el propio proyecto se había impuesto.

Qué haríamos distinto. Fijar el benchmark de random walk desde el diseño del experimento de supervivencia, no como corrección posterior a un resultado que ya emocionó a alguien. Dar al experimento cero de la semana 2 autoridad real para bajar el nivel de inversión del resto del programa, no tratarlo como paso protocolario. Congelar un lockbox técnico desde el día 1 (últimos 3 meses del histórico, con una barrera de código que niegue el acceso, no solo una promesa) — la disciplina que se sabía necesaria pero nunca se convirtió en un candado real terminó cediendo bajo presión de calendario. Y, sobre todo: aceptar desde el principio que "no hay edge" era el resultado más probable, y diseñar el proyecto genuinamente para intentar refutar esa corazonada lo más rápido posible — no para perseguir un número hasta que se volviera positivo.

§16 · Cierre

La regla que gobierna todo lo anterior

Cada semana de este roadmap termina con una pregunta científica respondida, registrada en board.md, con un veredicto entre cuatro estados. Ninguna semana existe solo para escribir infraestructura — la infraestructura de cada semana es exactamente la mínima necesaria para responder su pregunta, ni una línea más. Si en algún punto una decisión mejora la arquitectura pero no acelera una respuesta sobre el mercado, la decisión correcta, bajo este documento, es no tomarla.
