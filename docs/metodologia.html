§0

Principio rector

El objetivo no es confirmar que hay un edge. Es diseñar el proceso más rápido posible para refutar esa hipótesis, y solo aceptarla si sobrevive el intento serio de refutación. Un resultado negativo, documentado con el mismo rigor que uno positivo, es un resultado completo — no una investigación fallida.

Cada decisión metodológica de este documento existe para resolver una tensión concreta entre dos fuerzas: la que empuja a encontrar algo (porque se invirtió tiempo, porque un patrón que "casi" cruza el umbral se siente como un descubrimiento) y la que exige que un hallazgo sobreviva escrutinio antes de gastar un solo dólar operándolo.

§1

Ciclo de vida de un experimento

1. Pre-registro hypothesis.md — hipótesis, benchmark, criterios, ANTES de tocar datos 2. Checklist previo §9 3. Ejecución run.py — carga → valida → features → test → reporte 4. Checklist posterior §10 5. Abogado del diablo §12 — obligatorio antes de aceptar cualquier resultado positivo 6. Registro obligatorio formato fijo, §11 — se escribe funcione o no el experimento 7. Nivel de evidencia §8 — asignación por regla, no por impresión 8. Actualizar board.md nunca a mano — reflejar exactamente el resultado 9. (Opcional) Profundizar §7 — el patrón "00XA-G" cuando el resultado agregado es ambiguo

Ninguna etapa se salta. Si el paso 1 no está completo, el paso 3 no se ejecuta.

§2

Pre-registro (hypothesis.md)

Cada hipótesis nueva empieza con un archivo que no se edita después de ver resultados:

# 00X · <nombre corto> ## Hipótesis Una frase falsable. ## Mecanismo esperado Por qué, en términos de microestructura/estadística/economía, se esperaría que esto sea cierto si lo es. ## Benchmark Contra qué se compara — nunca "cero" ni "50%" por defecto (§3). ## Criterio de éxito (escrito antes de ver datos) Números exactos: estadístico, umbral, tamaño de efecto mínimo. ## Criterio de abandono (escrito antes de ver datos) Igual de exacto que el de éxito. ## Qué resultado refutaría la hipótesis Una frase. ## Datos Universo, rango, resolución real disponible (declarar limitaciones). ## Test estadístico Cuál, y por qué ese y no otro (§5). ## Número de comparaciones Cuántas pruebas suma este experimento al conteo acumulado de FDR. ## Lockbox Confirmar explícitamente que no se toca el lockbox reservado (§6).

Regla dura: si después de correr el experimento se quiere cambiar cualquier campo, el experimento se descarta y se vuelve a pre-registrar como uno nuevo. No se edita el original.

§3

Diseño del benchmark — la lección central del proyecto

El error más caro que se puede cometer en este proyecto es comparar contra un benchmark trivial.

Precedente concreto: en 001_survival, la probabilidad cruda de que una vela conserve su lado a T=240s es 0.8712. Comparado contra 50%, eso parece un edge enorme. Pero cualquier random walk puro produce una curva de "supervivencia" ascendente con T — simplemente porque queda menos tiempo para que algo cambie. El benchmark correcto no es una constante: es una simulación del proceso nulo más parecido posible a los datos reales en todo lo que no es la hipótesis que se está probando.

Procedimiento general

  1. Identificar qué parámetros de nulo (volatilidad, tendencia, autocorrelación de corto plazo) podrían por sí solos producir un patrón parecido, sin que exista información real.
  2. Simular ese proceso nulo emparejado observación por observación con esos parámetros — no con un promedio global.
  3. Repetir la simulación N veces (Monte Carlo, 200-300 réplicas en la práctica de este proyecto) para obtener una distribución del estadístico bajo el nulo.
  4. Comparar el resultado real contra esa distribución completa (media + percentiles 2.5/97.5), nunca contra un solo número.
# lib/stats.py — patrón reutilizable para cualquier hipótesis que # necesite un benchmark "¿esto es más de lo que el ruido produciría?" def simulate_random_walk_sides(sigma, rng, t_list): steps = rng.normal(loc=0.0, scale=sigma.reshape(-1, 1), size=(len(sigma), len(t_list))) cum_log_return = np.cumsum(steps, axis=1) return np.sign(cum_log_return)
Regla de aplicación

Antes de escribir el test estadístico de cualquier hipótesis nueva: "¿qué produciría este mismo estadístico si la hipótesis fuera falsa pero todo lo demás sobre el mercado fuera idéntico?" Esa respuesta, simulada, es el benchmark. Si la respuesta es "50%" o "cero", probablemente se está subestimando el efecto trivial.

§4

Prevención de fuga de información (leakage)

Regla general

Antes de usar una variable X para explicar o segmentar una decisión tomada en el instante T, preguntar: ¿el cálculo de X usa algún dato con timestamp posterior a T? Si sí, X puede describir la vela a posteriori (valor diagnóstico) pero no puede usarse como variable de condicionamiento en tiempo real.

Caso real detectado y corregido en este proyecto

Durante la profundización 001A-G se auditó el propio reporte base de 001_survival y se encontró que el heatmap de régimen de volatilidad usaba realized_vol_1m_intracandle — calculada con las 5 sub-velas completas, incluida la última, que coincide con el cierre. Usar esa variable para clasificar "régimen" en una decisión de T=240s es fuga.

NecesidadVariable con fugaVariable corregidaPor qué no filtra
Régimen de volatilidadrealized_vol_1m_intracandleatr_14Rolling de velas ya cerradas
Forma interna hasta T=240spath_efficiency_ratiopath_efficiency_240 (solo sub-velas 1-4)Nunca toca la sub-vela 5
Tendencia local—trend_return_20 con close.shift(20)Estrictamente anterior

Nota: usar una variable "con fuga" dentro del motor de benchmark (como sigma de la simulación Monte Carlo) sigue siendo correcto — ahí no se usa como señal operable, se usa para emparejar volatilidad estadísticamente. La fuga solo importa cuando la variable se presenta implícitamente como algo conocible en tiempo real.

  • ¿Se calcula con datos estrictamente anteriores al instante de decisión?
  • Si incluye la vela actual, ¿qué sub-parte exactamente, y ya ocurrió antes del punto de decisión?
  • ¿Se usa como condicionamiento o como variable diagnóstica? Declararlo explícitamente.
§5

Motor estadístico

5.1 — Intervalos de confianza: Wilson

Para cualquier proporción, usar el intervalo de Wilson — más estable que el normal aproximado cerca de 0 o 1.

lo, hi = scipy_stats.binomtest(successes, n).proportion_ci( confidence_level=1 - alpha, method="wilson" )

5.2 — Por qué NO usar un permutation test de shuffle aquí

Barajar la etiqueta final manteniendo la variable de "lado en T" rompería la continuidad del camino de precio que genera el artefacto mecánico del §3 — hasta datos de ruido puro parecerían "significativos" frente a ese null incorrecto. Por eso se usa Monte Carlo paramétrico (simular el proceso), no shuffle. La elección del test depende de qué mecanismo genera el patrón que se quiere descartar, nunca es genérica.

5.3 — Corrección por comparaciones múltiples (FDR, Benjamini-Hochberg)

def benjamini_hochberg(pvals): n = len(pvals) order = np.argsort(pvals) ranked = pvals[order] q_raw = ranked * n / (np.arange(n) + 1) q_monotone = np.minimum.accumulate(q_raw[::-1])[::-1] q = np.empty(n) q[order] = np.clip(q_monotone, 0, 1) return q

El conteo es acumulado por hipótesis, no por corrida. La decisión final usa el q-valor, nunca el p-valor crudo, salvo en la primerísima observación exploratoria (Nivel 1). Con 300 réplicas Monte Carlo, la resolución mínima del p-valor empírico es 1/300≈0.0033 — si un resultado queda cerca del umbral post-FDR, subir las réplicas antes de decidir.

5.4 — Significancia vs. relevancia

significativo = (q_valor < ALPHA) and (effect_size >= EFFECT_SIZE_MIN)

ALPHA y EFFECT_SIZE_MIN se fijan en el pre-registro a partir de razonamiento económico, nunca copiados mecánicamente de otra hipótesis.

§6

Lockbox y validación temporal

  • Se reserva un tramo final del histórico (90 días en 001_survival) antes de cualquier análisis exploratorio.
  • Prohibido leer, describir o graficar ese tramo hasta la validación final de una hipótesis en Nivel 3.
  • Todo desarrollo exploratorio (incluida la profundización §7) ocurre solo sobre el conjunto de desarrollo.
  • Mirarlo una sola vez lo quema — se reserva uno nuevo y se documenta el incidente.
§7

Profundización de una hipótesis (patrón "00XA-G")

Cuando el resultado agregado es ambiguo (Nivel 1), antes de abandonar o subir de nivel: ¿el efecto está distribuido uniformemente o concentrado en un contexto específico?

7.1 — Ejes de estratificación estándar

EjeCómo se construye sin fugaEjemplo
RégimenCuantiles de una variable conocida antes de la decisiónatr_14 en terciles (001A)

7.2 — Por qué las interacciones se preespecifican

Elegir la celda "más prometedora" después de ver todos los resultados es el sesgo de Texas sharpshooter. La disciplina es escribir, antes de correr nada, la motivación teórica de cada celda que se va a probar.

7.3 — Veredicto automático

n_significativos = comparaciones que sobreviven FDR + umbral de efecto n_tested = comparaciones totales evaluadas si n_significativos == 0 → permanece en observación si 0 < n_significativos < 0.5 × n_tested → dividir en hipótesis más específica si n_significativos >= 0.5 × n_tested → sube de nivel (con reservas de §8)

El umbral 0.5 es un default razonable, no un dogma — pero debe fijarse antes de ver los resultados de la profundización.

§8

Niveles de evidencia

NivelNombreCriterio objetivo de entrada
0Ideahypothesis.md completo, ningún dato tocado
1ObservaciónChecklist previo completo, primera corrida documentada, sin FDR todavía
2Hipótesis apoyadaSobrevive test individual con FDR preliminar, en una ventana/régimen
3Evidencia robustaEstable en ≥3 regímenes/años independientes, sobrevive el abogado del diablo
4Edge validadoTodo lo anterior + EV neto positivo, validado sobre el lockbox nunca tocado

Subir de nivel exige el criterio objetivo del siguiente — nunca por consenso subjetivo. Bajar es inmediato ante cualquier violación en auditoría. Techo estructural: con ~21 meses continuos de historia, ningún resultado puede superar Nivel 2 — Nivel 3 exige 3 regímenes macro independientes que ese rango todavía no cubre.

§9-10

Checklists previo y posterior

Antes de ejecutar

  • Hipótesis falsable, con mecanismo esperado
  • Criterio de éxito exacto (número, no impresión)
  • Criterio de abandono exacto
  • Benchmark definido explícitamente (§3)
  • Test estadístico elegido y justificado antes de ver datos (§5.2)
  • Tamaño de efecto mínimo económicamente relevante
  • Número de comparaciones que suma al conteo FDR
  • Ventana de datos exacta, sin tocar el lockbox
  • Ruta de resultados fijada de antemano
  • Resultado que refutaría la hipótesis, en una frase

Después de ejecutar

  • ¿Hay leakage? (§4, sobre cada feature nueva)
  • ¿Hay overfitting? (brecha in-sample vs. walk-forward)
  • ¿Se recalculó el q-valor con el conteo FDR actualizado?
  • ¿Hay estabilidad temporal, o se declaró el techo de nivel?
  • ¿El efecto es económicamente relevante, no solo estadístico?
  • ¿Se comparó contra el benchmark correcto, no uno trivial?
  • ¿Sobrevive el abogado del diablo?
  • ¿Se documentó en el registro obligatorio, funcione o no?
  • ¿Se actualizó board.md?

§12

El abogado del diablo

Sección obligatoria antes de aceptar cualquier resultado positivo, titulada exactamente: "Las cinco mejores razones por las que este resultado probablemente sea falso." Específicas al resultado concreto, nunca genéricas.

  1. ¿El benchmark pudo estar mal calibrado (supuestos que no se cumplen exactamente)?
  2. ¿La resolución de datos disponible es más gruesa que la ideal?
  3. ¿El periodo cubierto cruza suficientes regímenes?
  4. ¿La resolución del test (réplicas, tamaño de muestra) es suficiente para el umbral aplicado?
  5. ¿El número de comparaciones realizadas está correctamente contado en el FDR?

§15

Árboles de decisión

Señal interesante aparece

¿Pre-registrada antes de verla? No: re-registrar y probar en datos frescos. Sí → ¿sobrevive checklist posterior? No: "evidencia débil", no se borra. Sí → ¿sobrevive abogado del diablo? No: degradar un nivel. Sí → ¿económicamente relevante? No: tope Nivel 2. Sí → subir de nivel y actualizar board.md.

Señal desaparece

¿Fue en el lockbox o en validación regular? Lockbox: aceptar y degradar a "rechazada" sin excepción. Regular → ¿cambió el régimen? Sí: "evidencia condicionada a régimen", tope Nivel 3. No: sospechar leakage, re-auditar §4 y §10.

Señal inestable entre folds

¿Muestra suficiente por fold? No: "requiere más datos" (Nivel 2). Sí → ¿correlaciona con régimen? Sí: reformular como hipótesis condicional. No: tratar como ruido, degradar con fecha de revisión futura.

Contradice otra hipótesis aceptada

¿Mismo periodo/definición de datos? No: diferencia de muestra, documentar sin forzar reconciliación. Sí → ¿alguna no pasó el checklist completo? Sí: esa es la sospechosa. No (ambas limpias): tratar la contradicción como información real sobre un régimen no identificado.

§16

Estructura de una hipótesis nueva

research/00X_<nombre>/ ├── hypothesis.md # plantilla de §2 ├── run.py # carga → valida → features → test → reporte ├── results/ │ ├── report.html # generado por lib/report.py::make_report │ ├── figures/ │ └── summary.json # estado, nivel, métricas clave — alimenta board.md └── README.md # notas del investigador, actualizadas cada corrida

Regla de independencia: ninguna carpeta de research/ importa de otra. Solo de lib/ (funciones puras) y de data/ (solo lectura).

§17

Plantillas listas para copiar

Umbral de tendencia sin fuga (banda muerta basada en ruido)

log_close = np.log(df["close"].to_numpy()) lookback = 20 # ajustar por hipótesis trend_return = log_close - np.concatenate([np.full(lookback, np.nan), log_close[:-lookback]]) r1 = np.diff(log_close, prepend=np.nan) sigma_1bar = np.nanstd(r1) deadband = sigma_1bar * np.sqrt(lookback) trend = np.where(np.isnan(trend_return), "sin_dato", np.where(trend_return > deadband, "alcista", np.where(trend_return < -deadband, "bajista", "lateral")))

Prueba de un subgrupo contra su propio benchmark Monte Carlo

def run_group_test(df_subset, label, n_reps=300, seed=42, decision_t=240): if len(df_subset) < 300: return {"label": label, "potencia_insuficiente": True} curve = survival_curve(df_subset) bench, p_values = random_walk_benchmark(df_subset, n_reps=n_reps, seed=seed) real = curve.set_index("T_seconds").loc[decision_t, "p_same_side_at_close"] bmk = bench.set_index("T_seconds").loc[decision_t, "p_same_side_benchmark_mean"] return {"label": label, "n": len(df_subset), "effect_size": real - bmk, "p_empirico": p_values[decision_t]}
§18

Checklist definitivo — resumen ejecutivo

Antes de declarar cualquier hipótesis confirmada, verificar las siete preguntas que resumen todo este documento:

  1. ¿El benchmark es una simulación del proceso nulo correcto, no una constante trivial?
  2. ¿Ninguna variable de condicionamiento usa datos posteriores al punto de decisión?
  3. ¿El p-valor está corregido por el conteo total de comparaciones del proyecto?
  4. ¿El efecto es económicamente relevante, no solo estadísticamente distinguible de cero?
  5. ¿Es estable en ≥3 regímenes/años, o se declaró el techo de nivel por falta de cobertura?
  6. ¿Sobrevivió un intento serio y específico de encontrar por qué es falso?
  7. ¿Está documentado en el formato fijo, con el lenguaje correcto, funcione o no?

Si alguna respuesta es no, el nivel de evidencia se limita en consecuencia — no se fuerza una conclusión.
