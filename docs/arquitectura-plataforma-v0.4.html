§00 · Resumen

Una plataforma, no un proyecto

El objetivo de este RFC es que agregar la hipótesis número 300 cueste lo mismo, en términos de fricción arquitectónica, que agregar la número 3: un archivo nuevo en un directorio de plugins, nunca una modificación del núcleo.

El diseño se apoya en un principio simple aplicado con disciplina: todo lo que puede ser un archivo declarativo (config, registro, metadata) debe serlo, y todo lo que puede ser una clase descubrible por un registro (feature, experimento, modelo) debe serlo. El núcleo del sistema —ingestión, validación, resolución de dependencias, motor de reportes— no cambia nunca por una hipótesis nueva; solo crece el catálogo de plugins.

Este documento no entrena nada ni ejecuta nada. Es, deliberadamente, solo la arquitectura — y termina, por petición explícita, cuestionándose a sí misma: la última sección diseña la versión mínima que un solo ingeniero construiría en seis meses, que es deliberadamente más pequeña que todo lo descrito antes de ella.

§01 · Filosofía del sistema

Cuatro principios, sin excepciones silenciosas

Modularidad

Una hipótesis nueva = un archivo nuevo en experiments/library/. Si implementarla exige tocar core/, el core está mal diseñado, no la hipótesis.

Reproducibilidad

Todo resultado se ata a (commit de git, hash de dataset, hash de config, hash de entorno). Sin las cuatro claves, un resultado no es reproducible — es una anécdota.

Escalabilidad

El coste marginal de la feature 1000 o el experimento 500 debe ser igual al de la segunda. Si crece, el registro/caché está mal diseñado.

Responsabilidad única

Ningún módulo decide algo que le corresponde a otro: Model Training nunca decide particionado temporal; Calibration nunca conoce qué modelo generó las probabilidades.

§02 · Organización del proyecto

Separar código, configuración, datos y resultados

La propuesta del enunciado (todo como hermanos de data/) mezcla código con artefactos. Se corrige con una distinción de dos ejes: código instalable (src/, versionado en git, sin datos) vs. datos y resultados (data/, runs/, reports/, gitignored, versionados por content-hash, no por git). Es la misma disciplina que separa código de infraestructura en cualquier sistema serio: lo que cambia porque alguien programó no debe vivir junto a lo que cambia porque un experimento corrió.

research-platform/ ├── src/research_platform/ # único paquete importable — "el core" │ ├── ingestion/ # clientes Binance REST/WS, escritores Parquet │ ├── cleaning/ # dedupe, forward-fill de contexto, marcado de gaps │ ├── validation/ # validadores registrables, ValidationReport │ ├── features/ │ │ ├── core/ # Feature base, registro, resolución de dependencias, cache │ │ └── library/ # implementaciones: atr.py, path_efficiency.py, ofi.py... │ ├── datasets/ # DatasetBuilder, DatasetSpec, catálogo │ ├── experiments/ │ │ ├── core/ # Experiment base, registro, runner │ │ └── library/ # survival.py, markov.py, entropy.py, ofi.py... │ ├── statistics/ # tests puros: bootstrap, permutation, ADF, KPSS, BDS, Hurst... │ ├── models/ │ │ ├── adapters/ # wrapper uniforme por librería (sklearn, LightGBM...) │ │ └── registry/ # catálogo de modelos entrenados │ ├── calibration/ # Platt, Isotonic, reliability diagrams, ECE │ ├── meta_labeling/ # independiente — consume predicciones, no modelos │ ├── survival/ # independiente — Kaplan-Meier, Cox, hazard │ ├── visualization/ # registro de renderers por tipo de resultado │ ├── reporting/ # motor de templates HTML │ ├── versioning/ # hashing de contenido, run registry (SQLite) │ ├── cli/ # comandos delgados sobre el core │ └── common/ # config loader (Pydantic+YAML), logging, excepciones ├── config/ # YAML — activos, experimentos, datasets, plataforma ├── data/ # gitignored — raw/ processed/ features/ datasets/ external/ ├── runs/ # OUTPUT versionado — experiments/<nombre>/<run_id>/, models/… ├── reports/ # HTML generado, enlazado desde runs/ ├── tests/ # unit/ integration/ statistical/ regression/ ├── docs/ # handwritten + autogenerada desde el registro ├── scripts/ # mantenimiento puntual, nunca importado por src/ └── notebooks/ # exploración — nunca dependencia del paquete

Regla dura: nada dentro de src/research_platform/ importa nada de notebooks/ o scripts/ — la dirección de dependencia es siempre notebooks/scripts → paquete, nunca al revés. Es lo que evita que "un notebook lleno de scripts" sea, en la práctica, el verdadero núcleo del sistema.

§03 · Arquitectura por módulos

Responsabilidad, interfaz, entradas, salidas, dependencias

Data Ingestion

Responsabilidad
Obtener datos crudos de Binance (REST klines históricos; WebSocket para aggTrade/forceOrder/funding en vivo) y escribirlos sin transformar.
Interfaces
Ingestor.fetch(symbol, timeframe, start, end) -> RawBatch; Ingestor.stream(symbol, channel) -> Iterator[Event]. Cada fuente es una subclase registrada.
Entradas
Símbolo, timeframe, rango de fechas, o modo streaming.
Salidas
Parquet en data/raw/<fuente>/<symbol>/year=/month=/, inmutable, append-only.
Dependencias
Ninguna interna — es la hoja del árbol de dependencias del sistema.

Cleaning

Qué corrige
Solo tres cosas lícitas: elimina duplicados exactos por clave primaria; hace forward-fill de campos de contexto de baja frecuencia (funding, OI) cuando el timestamp de alineación es correcto; marca (nunca interpola) gaps de velas.
Qué NO hace
Nunca inventa un precio OHLC. Un gap se marca para que Validation y el investigador decidan — interpolar silenciosamente sería fabricar información.
Detección de errores
Delega enteramente a Validation; actúa solo sobre lo clasificado como "corregible automáticamente".

Validation

Conjunto fijo de validadores (no un registro plugin — el catálogo de chequeos de calidad de datos no crece al ritmo de las hipótesis):

ValidadorDetectaSeveridad
DuplicateTimestampValidatorFilas con misma clave (symbol, open_time).Bloqueante
GapValidatorVelas faltantes respecto a la grilla temporal esperada.Warning (embargo, no bloqueo)
MonotonicTimeValidatorTimestamps no estrictamente crecientes dentro de una partición.Bloqueante
OHLCConsistencyValidatorhigh < max(open,close,low), low > min(...), volumen negativo.Bloqueante
OutlierValidatorRetorno > N desviaciones robustas de la mediana móvil.Warning
TimezoneValidatorTimestamps naive (sin tz) o no normalizados a UTC.Bloqueante

Cada corrida produce un ValidationReport escrito en data_quality_log (SQLite, §05). Los issues bloqueantes detienen el pipeline para esa partición; los warning se registran y quedan disponibles para exclusión/embargo por fold en la validación temporal (coherente con v0.1 §08), nunca se silencian.

Feature Engineering

Registro
Decorador @register_feature; auto-descubrimiento al escanear features/library/.
Dependencias
Cada Feature declara depends_on: list[str] (columnas crudas u otras features); un resolutor calcula el orden de ejecución.
Evitar recálculo
Se recalcula solo si: no existe en caché, su version cambió, o alguna dependencia cambió de versión — invalidación por hash de versión, no por timestamp de archivo.
Caché
Cada feature calculada se persiste en data/features/<nombre>/<version>/...parquet; el FeatureStore resuelve qué versión sirve cada solicitud.

Detalle completo del contrato de clase en §06.

Dataset Builder

Responsabilidad
Combinar features por nombre+versión, según un DatasetSpec declarado en YAML, en un dataset materializado.
Interfaces
DatasetBuilder.build(spec: DatasetSpec) -> Dataset.
P0 vs. completo
Mismo mecanismo, distinto YAML: config/datasets/p0_core.yaml lista solo las features P0 del diccionario de datos (Fase 2); full.yaml las lista todas. No hay lógica especial para "modo P0".
Salidas
data/datasets/<nombre>/<content_hash>/ + metadata (spec usada, nº filas, rango temporal).

Experiment Engine — el corazón

Arquitectura tipo plugin: Experiment abstracta con lifecycle load_data → execute → analyze → generate_report → save_results. Un experimento concreto solo implementa execute() y analyze(); generate_report() y save_results() tienen implementación por defecto en la base (usan Visualization/Reporting/Versioning automáticamente). Registro vía @register_experiment("survival"), descubierto al arrancar el CLI. Cada experimento declara required_columns; el engine valida esto antes de ejecutar, para fallar rápido en vez de a mitad de un cálculo costoso.

Detalle completo del contrato en §07.

Statistical Engine

Contenido
Runs test, variance ratio, BDS, ADF, KPSS, Hurst (R/S y DFA), información mutua, bootstrap (i.i.d. y por bloques), permutation test genérico.
Diseño
Funciones puras test(data, **params) -> TestResult, sin estado, sin registro plugin — excepción deliberada a la regla general (ver autocrítica §14): el catálogo de tests estadísticos estándar no crece al ritmo de las hipótesis del negocio.

Visualization

Mecanismo
No es una librería de gráficas — es un registro de renderers por tipo de resultado. Cada TestResult/Analysis declara un plot_hint (p.ej. "kaplan_meier_curve", "reliability_diagram", "heatmap").
Consecuencia
Un experimento nuevo obtiene gráficas automáticas si reutiliza tipos de resultado existentes; solo registra un renderer nuevo si introduce un tipo de resultado genuinamente nuevo.

Reporting

Mecanismo
Motor de templates HTML que ensambla: hipótesis (metadata del experimento), metodología (config congelada), resultados (tablas de TestResult), gráficas (Visualization).
Límite honesto
Conclusiones y limitaciones no se auto-generan — son responsabilidad del método analyze() del investigador. Interpretar resultados es trabajo científico, no un template.
Salida
reports/<experimento>/<run_id>/index.html, enlazado desde un índice maestro.

Model Training

Contrato
ModelAdapter uniforme sobre sklearn/XGBoost/LightGBM/CatBoost/PyTorch: fit(X,y), predict_proba(X), get_params(), registrado igual que Features/Experiments.
Frontera estricta
Nunca decide partición temporal (walk-forward/CPCV vive en datasets/experiments) — solo entrena y predice.

Calibration

Independencia
Recibe (y_true, y_pred_proba) de cualquier fuente y devuelve un Calibrator (Platt/Isotonic) + diagnósticos (reliability diagram, ECE). No conoce ni le importa qué modelo generó las probabilidades — puede calibrar hasta una tabla de frecuencias histórica.

Meta-labeling

Independencia
Consume predicciones + historial de acierto/fallo de un modelo primario + features de confianza (v0.2 §05); entrena el clasificador secundario. Solo conoce la interfaz de predicción del primario, nunca su implementación.

Survival Analysis

Independencia
Kaplan-Meier, Cox, curvas de hazard sobre survival_states_5m (Fase 2). No depende de Model Training ni de Experiment Engine — usable standalone desde un notebook, consumido por SurvivalExperiment sin acoplamiento adicional.

Configuration

Mecanismo
Todo parámetro vive en YAML bajo config/, cargado y validado contra un schema (Pydantic) al arrancar.
Reproducibilidad
Cada run de experimento persiste una copia congelada de la configuración efectiva junto a sus resultados — nunca una referencia a un archivo que puede cambiar después.

CLI

Comandos delgados: resuelven config → llaman al módulo del core correspondiente → nunca contienen lógica de negocio (así el core es testeable sin pasar por CLI).

research build-dataset <spec> research compute-features [--only <names>] [--force-recompute] research validate <dataset> research run experiment <name> [--config <path>] [--dry-run] research run experiment survival --asset BTCUSDT --from 2020-01-01 --to 2024-01-01 research train <model> --dataset <name> research calibrate <run_id> research report latest research report <run_id> research list experiments|features|models|runs research docs build research doctor # salud del sistema: registro vs. archivos, config, caché research verify-repro <run_id> # reproduce un run y compara resultados bit a bit
§04 · Flujo completo del sistema

De la descarga al resultado versionado

  1. Descargar
    Ingestion escribe data/raw/, inmutable, particionado. No transforma nada.
  2. Validar
    Validation corre los 6 validadores; issues bloqueantes detienen la partición, warnings se registran en data_quality_log sin detener.
  3. Limpiar
    Cleaning aplica solo dedupe/forward-fill de contexto/marcado de gaps sobre lo que pasó validación.
  4. Construir features
    Feature Engineering resuelve el grafo de dependencias, calcula solo lo no cacheado, escribe data/features/.
  5. Construir datasets
    Dataset Builder combina features según un DatasetSpec YAML, materializa en data/datasets/<hash>/.
  6. Ejecutar experimentos
    Experiment Engine carga el dataset, corre execute()/analyze() del plugin, delega gráficas y reporte.
  7. Entrenar modelos
    Model Training (cuando corresponda) entrena sobre folds ya definidos por el experimento, nunca decide la partición.
  8. Evaluar
    Statistical Engine + Calibration producen métricas y diagnósticos, agnósticos del origen del modelo.
  9. Reportar
    Reporting ensambla el HTML final; conclusiones son responsabilidad del investigador, no del template.
  10. Versionar
    Versioning ata (commit git, hash dataset, hash config, hash entorno) al run_id, escribe en runs/, inmutable desde entonces.

El flujo es lineal en el caso feliz, pero cada paso puede ejecutarse de forma incremental e independiente vía CLI — no hace falta re-descargar ni re-validar todo para correr un experimento nuevo sobre datos ya procesados.

§05 · Bases de datos

Cada motor, justificado por su rol — no por defecto

MotorQué almacenaPor qué específicamente este motor
ParquetFuente de verdad primaria: klines crudos, features, datasets materializados, resultados tabulares de experimentos.Columnar, comprime bien, soporta tipos anidados (necesarios para candle_path_5m), formato nativo de DuckDB y Polars. Es la única capa que importa versionar por content-hash.
DuckDBNada permanente — motor de consulta embebido que abre los Parquet bajo demanda para joins, agregaciones y ventanas.Cero proceso servidor, cero estado propio que pueda desincronizarse de los archivos. Sustituye por completo la necesidad de un warehouse para el volumen de este proyecto (<2 GB, Fase 2 §14.4).
SQLiteMetadata operativa de baja escala y alta necesidad transaccional: catálogo de runs (run_id, config hash, estado, duración), registro de versiones de features/datasets/modelos, data_quality_log.Estas tablas son pequeñas (miles de filas) pero se escriben concurrentemente desde varios procesos (experimentos en paralelo) y necesitan ACID simple — algo que Parquet no da sin un motor tipo Delta/Iceberg, que sería sobre-ingeniería aquí. SQLite es correcto por el volumen bajo y la necesidad transaccional real, no por ser "la opción por defecto".
RedisNo incluido en el diseño base.Reservado como componente opcional y condicional, solo si se construye ingestión de streaming en vivo que necesite un buffer de baja latencia entre WebSocket y escritura por lotes. Hoy ese rol lo cumple una cola en archivo/proceso simple — no se justifica una pieza de infraestructura con estado propio para investigación por lotes.
Memoria (in-process)DataFrames Polars/Pandas durante el cálculo de una feature o experimento — nunca como almacén persistente.El dataset completo cabe en <2 GB; toda una sesión de investigación vive cómodo en la RAM de una laptop. Es una ventaja de diseño explícita: no hace falta cómputo distribuido durante años de crecimiento razonable.
Archivos planos (YAML/JSON/HTML)Configuración, metadata de features/experimentos versionada junto a su código, reportes.Su unidad natural de cambio es "un commit" o "un run", no "una fila" — no pertenecen a una base de datos.

§07 · Sistema de experimentos

Agregar una hipótesis debe doler lo menos posible

class Experiment(ABC): name: str hypothesis: str # texto humano — va directo al reporte required_columns: list[str] # valida antes de ejecutar, falla rápido def load_data(self, dataset: Dataset) -> pl.DataFrame: ... @abstractmethod def execute(self, data: pl.DataFrame) -> ExperimentResult: ... @abstractmethod def analyze(self, result: ExperimentResult) -> Analysis: ... def generate_report(self, analysis: Analysis) -> Report: ... # default: usa Visualization+Reporting def save_results(self, analysis: Analysis, report: Report) -> RunRecord: ... # default: usa Versioning

Un experimento nuevo (p. ej. EntropyExperiment) hereda de Experiment, implementa solo execute() y analyze(), y se registra con @register_experiment("entropy") en experiments/library/entropy.py. Nada más en el sistema cambia. El CLI lo descubre automáticamente al escanear el directorio; research doctor verifica que el número de archivos coincide con el número de entradas registradas, para atrapar errores de importación silenciosos.

§08 · Versionado

Reproducibilidad como contrato, no como buena intención

ArtefactoCómo se versiona
DatasetsHash del DatasetSpec YAML + hash de contenido de los Parquet resultantes; el nombre físico incluye el hash, nunca se sobreescribe.
Featuresversion declarada por el autor + hash automático del código fuente de compute() como chequeo de integridad — si el código cambió sin bump de versión, el sistema lo detecta y bloquea el uso silencioso de caché obsoleto.
ExperimentosCada run_id persiste: config congelada, hash del dataset usado, hashes de versión de cada feature, commit de git del código del experimento, resultados, y hash del lockfile de entorno.
ModelosIgual que experimentos, más el artefacto binario serializado con su propio hash.
ReportesInmutables una vez generados; un reporte nuevo es siempre un run_id nuevo, nunca una regeneración in-place.
ConfiguraciónVersionada en git como código; cada run guarda además una copia congelada, no una referencia mutable.
ResultadosParquet/JSON dentro de runs/<tipo>/<nombre>/<run_id>/, inmutables.

La unidad última de reproducibilidad es la tupla (git_commit, dataset_hash, config_hash, environment_lockfile_hash). research verify-repro <run_id> hace checkout del commit registrado, reconstruye el entorno del lockfile, y compara resultados bit a bit — el comando que convierte "reproducibilidad" de aspiración en algo que se puede fallar en CI.

§09 · Testing

Cuatro capas, cada una con un objetivo distinto

Unitarias

Cada Feature.compute() contra inputs sintéticos de casos borde (volumen cero, gap, vela con un solo trade) — nunca contra datos reales, para que fallen por una razón, no por deriva de mercado.

Integración

Pipeline completo (ingestion→features→dataset→experimento→reporte) sobre un dataset sintético fijo, corre en segundos en CI.

Estadísticas

Verifican propiedades matemáticas, no solo ausencia de crash: una curva Kaplan-Meier debe ser monótona no creciente; un permutation test sobre ruido puro no debe rechazar H0 más del 5% de las veces (meta-test del propio framework); un modelo sintéticamente bien calibrado debe dar ECE≈0.

Regresión

Snapshots de resultados de experimentos de referencia (golden datasets); un cambio silencioso de resultado numérico tras un refactor requiere aprobación explícita para actualizar el snapshot.

§10 · Logging

Observabilidad operativa, distinta de la reproducibilidad

Logging estructurado (JSON lines) con run_id, module, event, duration_ms, level, metadata — cada cómputo de feature, cada run de experimento, cada build de dataset, cada error emite al mismo stream, agregable después con el mismo DuckDB del resto del sistema (log analytics sin herramienta adicional). Distinción explícita respecto a §08: versionado es sobre si un resultado es reproducible; logging es sobre si el sistema está sano y qué tan rápido corre.

§11 · Escalabilidad a 5 años

1000 features, 500 experimentos, 100 modelos, 20 activos, 15 timeframes, 10 TB

DimensiónPor qué no rompe el diseño
1000 featuresRegistro + FeatureGraph: el coste marginal de la feature 1001 es igual al de la segunda.
500 experimentosPlugin registry: el Experiment Engine no crece en complejidad, solo crece el directorio library/.
100 modelosModelAdapter uniforme + catálogo en SQLite: comparar 100 modelos es una query, no un ejercicio manual.
20 activos / 15 timeframesEl particionado Hive-style (symbol=/timeframe=/year=/month=) ya anticipa esto desde la Fase 2 — escalar es multiplicar particiones, no rediseñar el esquema.
10 TBPunto de decisión explícito: mover data/raw y data/features a almacenamiento objeto (S3-compatible), DuckDB leyendo remoto de forma nativa. Elegir Parquet+DuckDB desde el día 1 es lo que hace esa migración "de laptop a bucket" casi gratuita, sin reescribir ningún módulo — no hace falta un motor distribuido.
El riesgo real no es de cómputo

Con 1000 features y 500 experimentos, el cuello de botella deja de ser técnico y pasa a ser de descubribilidad: "¿esto ya existe?". De ahí que Documentación (§12) y un catálogo consultable (research list features --family microstructure) sean tan críticos como el motor de cómputo mismo.

§12 · Documentación automática

Derivada del registro, nunca escrita a mano por separado

Cada Feature/Experiment expone su propia metadata estructurada (nombre, hipótesis, dependencias, versión, autor); research docs build recorre el registro completo y genera páginas por entidad — mismo patrón que un autodoc de Sphinx, aplicado a features/experimentos en vez de a funciones sueltas. Consecuencia directa: la documentación nunca queda desactualizada respecto al registro real, porque no es un documento aparte, es una vista del código. Los reportes de experimentos (§Reporting) ya son documentación y quedan enlazados automáticamente al catálogo; cada dataset materializado genera una "ficha" (features incluidas, rango temporal, filas, hash) por la misma razón de trazabilidad.

§13 · Roadmap de implementación

Construir lo que desbloquea investigación real primero

  1. Semana 1
    Esqueleto del repo (layout src/), config loader (Pydantic+YAML), logging, CI con un smoke test. Cero features todavía.
  2. Semana 2
    Data Ingestion (klines 1m/5m) + escritor Parquet particionado.
  3. Semana 3
    Validation (los 6 validadores) + data_quality_log (SQLite) + Cleaning mínimo.
  4. Semana 4
    Core de Feature Engineering (clase base, registro, grafo, caché) — solo 2-3 features triviales para probar el mecanismo.
  5. Semana 5
    Features P0 reales (Fase 2 §02-§06: básica, temporal, volatilidad núcleo, microestructura, trayectoria). Aquí vive la mayor parte del trabajo de contenido.
  6. Semana 6
    Dataset Builder + p0_core.yaml + primer dataset materializado.
  7. Semana 7
    Statistical Engine + core de Experiment Engine; Visualization/Reporting en versión mínima (2-3 tipos de gráfica, template HTML básico).
  8. Semana 8
    SurvivalExperiment end-to-end: primer corte vertical completo del sistema, sobre la hipótesis de mayor prioridad de v0.2.
  9. Semana 9
    Segundo experimento (estructura interna / Markov) — valida que "agregar hipótesis = agregar módulo" realmente se sostiene sin tocar el core; se corrige la fricción que aparezca.
  10. Semana 10
    Endurecer Versioning (hashes de contenido, catálogo de runs, verify-repro).
  11. Semana 11
    Order flow (§09 Fase 2) + ingestión de funding/OI/liquidaciones + OFIExperiment.
  12. Semana 12
    Backfill de testing (las 4 capas) sobre todo lo construido — deliberadamente después de tener contenido real que proteger, no antes.
  13. Semana 13
    Model Training (adaptadores logística + un GBM) + Calibration — primera vez que se tocan modelos, coherente con la filosofía "dataset antes que modelo" de las tres fases previas.
  14. Semana 14
    Meta-labeling — condicional a que el modelo primario de la semana 13 muestre algo de señal.
  15. Semana 15
    Documentación automática + pulido de CLI + research doctor.
  16. Semana 16
    Buffer, endurecimiento, primer reporte comparativo entre experimentos, retro del propio roadmap.
§14 · Revisión crítica

Autocrítica de esta misma arquitectura

Sobre-ingeniería temprana del FeatureGraph

Diseñar resolución de dependencias tipo grafo general antes de tener 20 features es trabajo especulativo. Con <50 features, una lista ordenada declarada a mano con chequeo de dependencias ya alcanza. El grafo se justifica solo cuando aparezcan ciclos/dependencias complejas reales.

El Statistical Engine "sin registro" es una excepción no anunciada

Contradice la promesa de "modularidad total" en §01. Es una excepción deliberada y razonable, pero debe reconocerse explícitamente como tal — no esconderse detrás de "modularidad siempre".

Riesgo de fragmentación en Calibration/Meta-labeling/Survival

Tres módulos "completamente independientes" pueden derivar en tres micro-frameworks con convenciones distintas si no comparten los mismos tipos base (Result/Analysis) que usa Experiment Engine. Mitigación: forzar esa interfaz común desde el día 1, no como refactor posterior.

"Reporte automático" tiene un límite real

Conclusiones siguen requiriendo que el investigador escriba texto de interpretación — es honesto decirlo así en vez de prometer que el sistema "genera reportes sin código adicional" sin matices.

Versionado exhaustivo es fricción real al principio

Con un equipo de 1 persona, atar cada run a 4 hashes distintos es disciplina cara antes de que haya algo que de verdad reproducir. Riesgo concreto de abandono si es demasiado pesado desde el día 1 — ver §16.

SQLite como single writer bajo concurrencia real

Un batch nocturno de cientos de experimentos en paralelo puede convertir el catálogo de runs en un cuello de botella de escritura. Mitigable con modo WAL; si el equipo crece más allá de 1-2 personas ejecutando intensivamente en paralelo, es el primer componente a reconsiderar (candidato: Postgres liviano — explícitamente pospuesto, no parte de este diseño).

15 módulos "independientes" es mucha superficie para 1-3 personas

Riesgo real de que la arquitectura completa nunca termine de construirse. Esta tensión es exactamente la que resuelve §16.

Auto-descubrimiento por escaneo de archivos puede esconder bugs silenciosos

Un experimento con error de sintaxis simplemente no aparece en el registro, sin error claro. Mitigado por research doctor, pero el riesgo es real y debe vigilarse activamente, no darse por resuelto.

§16 · La pregunta final — versión mínima viable

Olvidando todo lo anterior: ¿qué construiría un CTO con 1 ingeniero y 6 meses?

Todo lo descrito en §01–§15 es correcto como visión a 2-3 años si el edge existe y el equipo crece. Como plan de los primeros seis meses con un solo ingeniero, es la arquitectura equivocada: sería construir la fábrica antes de saber si el producto se vende. La pregunta que de verdad importa en seis meses no es "¿la plataforma escala a 1000 features?" sino "¿hay algo ahí?" — y ese objetivo cambia radicalmente qué se construye primero.

Eliminar por completo (0 horas en 6 meses)

Registro plugin formal con auto-descubrimiento para Experiment/Feature — con 5-10 experimentos reales posibles en 6 meses, un import directo en un script basta. Soporte multi-modelo (7 adaptadores) — construir solo logística + un GBM, y solo cuando haya señal que lo justifique. Documentación automática — con equipo de 1, el README es la documentación. Meta-labeling como módulo — es una función, no una arquitectura, hasta que el primario muestre señal real. Redis, cualquier orquestador (Airflow/Dagster/Prefect), cualquier base de datos con proceso propio (Postgres). El FeatureGraph tipo DAG — una lista ordenada a mano alcanza con <50 features. Versionado exhaustivo de 4 hashes — sustituido por git commit + un run_manifest.json por experimento.

Construir primero (semanas 1-8)

Ingestion de klines 1m/5m + Validation básica (gaps, duplicados, consistencia OHLC) — sin esto no hay nada. Features P0 de la Fase 2 §05-§08 (microestructura núcleo + supervivencia) — el contenido, no la infraestructura. Un experimento real, de principio a fin: supervivencia del color (Kaplan-Meier + tabla de frecuencias), con reporte aunque sea un notebook exportado a HTML. Segundo experimento: estructura interna / shape clustering. Esto responde, en dos meses, la única pregunta que importa.

Simplificar

15 módulos → 5 paquetes reales: data (ingestion+cleaning+validation juntos, porque en la práctica se tocan a la vez), features, experiments (incluye statistics+visualization+reporting básico), models (incluye calibration cuando llegue el momento), common. CLI de 10+ comandos → 3 reales: build-dataset, run-experiment, report; el resto son funciones de Python llamadas desde notebook durante desarrollo activo. Testing de 4 capas → unitarios solo para cálculos fáciles de equivocar silenciosamente (path efficiency, occupation time, hazard) + un único test de integración end-to-end.

Posponer más allá de los 6 meses

Todo lo anterior, salvo que las semanas 1-8 muestren señal real: model training completo, calibración formal, meta-labeling arquitectónico, y — esto es lo más importante de admitir — el objetivo mismo de "plataforma reutilizable para proyectos futuros". Ese objetivo es precisamente el primero que hay que sacrificar, porque optimiza para un futuro hipotético en vez de para la velocidad de aprendizaje inmediata.

Veredicto del CTO

La arquitectura de este documento (§01–§15) es la correcta para un fondo que ya sabe que tiene un edge y necesita sostener la investigación de un equipo durante años. La arquitectura correcta para los primeros seis meses de un equipo de una persona es aproximadamente el 20% de este documento: ingestión, validación, un puñado de features de altísima prioridad, y dos experimentos completos de punta a punta. Todo lo demás —registro de plugins, cuatro capas de versionado, documentación automática, quince módulos independientes— es exactamente el tipo de trabajo que se siente productivo y no mueve la aguja de la única pregunta que importa: ¿existe un edge real, sí o no? Se construye cuando la respuesta ya es "sí" y hay que sostenerlo — no antes.
